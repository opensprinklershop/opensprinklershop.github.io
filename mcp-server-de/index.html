MCP Server — KI-Integration für OpenSprinkler

Der OpenSprinkler MCP (Model Context Protocol) Server ermöglicht es KI-Assistenten wie GitHub Copilot, Claude und ChatGPT, dein Bewässerungssystem direkt abzufragen und zu steuern.

Zwei Betriebsmodi

1. Eingebauter Firmware-MCP (nur ESP32)

Die ESP32-Firmware enthält einen eingebauten MCP-HTTP-Endpunkt direkt auf dem Gerät:

POST http://<controller-ip>/mcp
Content-Type: application/json

Vorteile: Kein Node.js nötig, kein separater Prozess — die KI kommuniziert direkt mit dem Controller via HTTP (Streamable-HTTP-Transport, JSON-RPC 2.0).

Verfügbare Tools (10): get_all, get_controller_variables, get_options, get_stations, get_station_status, get_programs, get_debug, manual_station_run, change_controller_variables, pause_queue

2. Externer Node.js MCP-Server (alle Plattformen)

Der Node.js-Server unter tools/mcp-server/ stellt die komplette REST-API (40+ Tools) via stdio-Transport bereit.

Vorteile: Vollständige Abdeckung aller API-Endpunkte inklusive Sensoren, Programme, Monitore und Anpassungen.

Einrichtung

Voraussetzungen

  • Node.js ≥ 18
  • Ein erreichbarer OpenSprinkler-Controller im Netzwerk

Installation

cd tools/mcp-server
npm install
npm run build

Konfiguration in VS Code (GitHub Copilot)

In .vscode/mcp.json einfügen:

{
  "servers": {
    "opensprinkler": {
      "type": "stdio",
      "command": "node",
      "args": ["tools/mcp-server/dist/index.js"],
      "env": {
        "OS_BASE_URL": "http://<CONTROLLER-IP>",
        "OS_PASSWORD_HASH": "<MD5-HASH>"
      }
    }
  }
}

Konfiguration in Claude Desktop

In ~/.config/claude/claude_desktop_config.json einfügen:

{
  "mcpServers": {
    "opensprinkler": {
      "command": "node",
      "args": ["/vollstaendiger/pfad/zu/tools/mcp-server/dist/index.js"],
      "env": {
        "OS_BASE_URL": "http://<CONTROLLER-IP>",
        "OS_PASSWORD_HASH": "<MD5-HASH>"
      }
    }
  }
}

Passwort-Hash berechnen

echo -n "dein_admin_passwort" | md5sum | awk '{print $1}'

KI-gestützte Programmerstellung

Ein KI-Assistent kann vollständige Bewässerungsprogramme aus natürlichsprachlichen Anfragen erstellen.

So funktioniert es

  1. Die KI ruft get_stations auf → lernt Stationsnamen und Anzahl
  2. Die KI ruft get_options auf → lernt Zeitzone, Sonnenauf-/untergangseinstellungen
  3. Die KI ruft get_programs auf → sieht bestehende Programme (Konfliktvermeidung)
  4. Die KI ruft change_program mit pid=-1 auf → erstellt ein neues Programm

Programm-Kodierung

Das change_program-Tool verwendet dieses Format:

v=[flag, days0, days1, [start0,start1,start2,start3], [dauer0,dauer1,...]]

Flag-Bits

Bit Bedeutung
0 Aktiviert (1=ein)
1 Wetteranpassung verwenden (1=ja)
2-3 Einschränkung: 0=keine, 1=ungerade Tage, 2=gerade Tage
4-5 Zeitplantyp: 0=wöchentlich, 1=einmalig, 2=monatlich, 3=Intervall
6 Startzeit-Typ: 0=wiederholend, 1=feste Zeiten
7 Datumsbereich aktivieren

Tage-Kodierung

  • Wöchentlich: days0 = Wochentag-Bitmaske (Bit0=Mo, Bit1=Di, … Bit6=So)
  • Intervall: days0 = Rest, days1 = Intervall in Tagen
  • Monatlich: days0 = Tag des Monats (1–31, 32=letzter Tag)

Startzeit

  • Minuten ab Mitternacht (0–1439)
  • Sonnenaufgang: Bit14 gesetzt, Bit12=Vorzeichen, Bits 0–11=Offset
  • Sonnenuntergang: Bit13 gesetzt, Bit12=Vorzeichen, Bits 0–11=Offset

Beispiel

Benutzer sagt: „Bewässere den Rasen (Station 0) und die Blumenbeete (Station 1) jeden Morgen um 6 Uhr, 15 und 20 Minuten, Mo–Fr, mit Wetteranpassung"

{
  "pid": -1,
  "name": "Morgenbewässerung",
  "v": "[67,31,0,[360,-1,-1,-1],[900,1200,0,0,0,0,0,0]]"
}
  • flag=67 (aktiviert + Wetter + feste Startzeiten)
  • days0=31 (Mo–Fr)
  • start0=360 (6:00 Uhr)
  • dauer0=900 (15 Min), dauer1=1200 (20 Min)

Sensorbasierte Programmanpassungen

Bewässerungsdauer automatisch anhand von Live-Sensorwerten skalieren.

Anpassungstypen

Typ Name Verhalten
1 LINEAR Lineare Interpolation: Sensor min→max wird auf Faktor1→Faktor2 abgebildet
2 DIGITAL_MIN Wenn Sensor ≤ min: Faktor1, sonst: Faktor2
3 DIGITAL_MAX Wenn Sensor ≥ max: Faktor2, sonst: Faktor1
4 DIGITAL_MINMAX Wenn Sensor ≤ min ODER ≥ max: Faktor1, sonst: Faktor2

Lineare Formel (Typ 1)

Die Faktorwerte sind Bewässerungsmultiplikatoren (1.0 = 100%, 0.5 = 50%, 2.0 = 200%).

Wenn Faktor1 > Faktor2 (abnehmend): Ergebnis = (max - Sensor) / (max - min) × (Faktor1 - Faktor2) + Faktor2

Wenn Faktor1 < Faktor2 (zunehmend): Ergebnis = (Sensor - min) / (max - min) × (Faktor2 - Faktor1) + Faktor1

Beispiel: Weniger Bewässerung bei feuchterem Boden

{
  "nr": 1,
  "type": 1,
  "sensor": 1,
  "prog": 0,
  "min": 10,
  "max": 90,
  "factor1": 2.0,
  "factor2": 0.0,
  "name": "Feuchtigkeitsanpassung"
}

Bei 10% Feuchtigkeit → 200% Bewässerung. Bei 50% → 100%. Bei 90% → 0% (überspringen).

Beispiel: Mehr Bewässerung bei höherer Temperatur

{
  "nr": 2,
  "type": 1,
  "sensor": 2,
  "prog": 0,
  "min": 15,
  "max": 40,
  "factor1": 0.5,
  "factor2": 1.5,
  "name": "Temperaturanpassung"
}

Bei 15°C → 50% Bewässerung. Bei 27,5°C → 100%. Bei 40°C → 150%.

Beispiel: Regen-Abschaltung (Digital)

{
  "nr": 3,
  "type": 2,
  "sensor": 3,
  "prog": 0,
  "min": 5,
  "factor1": 0,
  "factor2": 1,
  "name": "Regen-Stopp"
}

Regen ≤ 5mm → Bewässerung stoppen (Faktor 0). Regen > 5mm → Normalbetrieb.

Monitor-System

Monitore überwachen Sensorwerte und lösen Programme basierend auf Bedingungen aus oder blockieren sie.

Monitor-Typen

Typ Name Beschreibung
1 MIN Auslösen wenn Sensor ≤ Schwellwert
2 MAX Auslösen wenn Sensor ≥ Schwellwert
3 SENSOR12 Digitalen Regen-/Bodensensor lesen
4 SET_SENSOR12 Digitalen OS-Sensor setzen
10 AND Logisches UND von bis zu 4 Monitoren
11 OR Logisches ODER von bis zu 4 Monitoren
12 XOR Logisches XOR von bis zu 4 Monitoren
13 NOT Logisches NICHT eines Monitors
14 TIME Zeitfenster (Wochentag + Zeitbereich)
100 REMOTE Monitor auf einem anderen OpenSprinkler

MIN/MAX-Parameter (Typen 1, 2)

Parameter Typ Beschreibung
value1 double Schwellwert
value2 double Sekundärwert (Hysterese)

Logik-Gatter-Parameter (Typen 10, 11, 12)

Parameter Typ Beschreibung
monitor1–monitor4 uint16 Monitor-IDs (bis zu 4)
invers1–invers4 0/1 Logik pro Eingang invertieren

TIME-Parameter (Typ 14)

Parameter Typ Beschreibung
from uint16 Startzeit als HHMM (z.B. 0630 = 6:30)
to uint16 Endzeit als HHMM (z.B. 1800 = 18:00)
wdays uint8 Wochentag-Bitmaske (Bit0=Mo … Bit6=So)

Beispiel: Nur bewässern wenn trocken UND morgens

Monitor A (MIN):  sensor=Feuchtigkeit, value1=30 → löst aus wenn <30%
Monitor B (TIME): from=0600, to=1000, wdays=127 → 6–10 Uhr täglich
Monitor C (AND):  monitor1=A, monitor2=B → Programm nur wenn beide erfüllt

Beispiel: Frost-/Hitzeschutz

{
  "nr": 10,
  "type": 4,
  "sensor": 2,
  "prog": 0,
  "min": 3,
  "max": 35,
  "factor1": 0,
  "factor2": 1,
  "name": "Frost-Hitzeschutz"
}

Unter 3°C oder über 35°C → Bewässerung stoppen. Sonst → Normalbetrieb.

MCP-Ressourcen

Der MCP-Server stellt zwei dynamische Ressourcen bereit, die KI-Assistenten für Kontext lesen können:

Ressource URI Inhalt
api-overview opensprinkler://api-overview Vollständige API-Referenz: Programmkodierung, Sensortypen, Anpassungen, Monitore
controller-summary opensprinkler://controller-summary Live-Daten: Firmware, Stationen, Programme, Status, WiFi

Diese Ressourcen geben der KI alle Dokumentation, die sie braucht, um das System zu verstehen und Programme zu erstellen, Sensoren zu konfigurieren sowie Anpassungen und Monitore korrekt einzurichten.
