MCP Server — AI Integration for OpenSprinkler

The OpenSprinkler MCP (Model Context Protocol) server enables AI assistants like GitHub Copilot, Claude, and ChatGPT to directly query and control your irrigation system.

Two Operating Modes

1. Built-in Firmware MCP (ESP32 only)

The ESP32 firmware includes a built-in MCP HTTP endpoint directly on the device:

POST http://<controller-ip>/mcp
Content-Type: application/json

Advantages: No Node.js required, no separate process — the AI communicates directly with the controller via HTTP (Streamable-HTTP transport, JSON-RPC 2.0).

Available Tools (10): get_all, get_controller_variables, get_options, get_stations, get_station_status, get_programs, get_debug, manual_station_run, change_controller_variables, pause_queue

2. External Node.js MCP Server (all platforms)

The Node.js server at tools/mcp-server/ provides the full REST API (40+ tools) via stdio transport.

Advantages: Complete coverage of all API endpoints including sensors, programs, monitors, and adjustments.

Setup

Prerequisites

  • Node.js ≥ 18
  • An accessible OpenSprinkler controller on your network

Installation

cd tools/mcp-server
npm install
npm run build

Configuration in VS Code (GitHub Copilot)

Add to .vscode/mcp.json:

{
  "servers": {
    "opensprinkler": {
      "type": "stdio",
      "command": "node",
      "args": ["tools/mcp-server/dist/index.js"],
      "env": {
        "OS_BASE_URL": "http://<CONTROLLER-IP>",
        "OS_PASSWORD_HASH": "<MD5-HASH>"
      }
    }
  }
}

Configuration in Claude Desktop

Add to ~/.config/claude/claude_desktop_config.json:

{
  "mcpServers": {
    "opensprinkler": {
      "command": "node",
      "args": ["/full/path/to/tools/mcp-server/dist/index.js"],
      "env": {
        "OS_BASE_URL": "http://<CONTROLLER-IP>",
        "OS_PASSWORD_HASH": "<MD5-HASH>"
      }
    }
  }
}

Computing the Password Hash

echo -n "your_admin_password" | md5sum | awk '{print $1}'

AI-Guided Program Creation

An AI assistant can create complete irrigation programs by understanding natural language requests.

How It Works

  1. The AI calls get_stations to learn station names and count
  2. The AI calls get_options to learn timezone, sunrise/sunset settings
  3. The AI calls get_programs to see existing programs (avoid conflicts)
  4. The AI calls change_program with pid=-1 to create a new program

Program Encoding

The change_program tool uses this format:

v=[flag, days0, days1, [start0,start1,start2,start3], [dur0,dur1,...]]

Flag Bits

Bit Meaning
0 Enable (1=on)
1 Use weather adjustment (1=yes)
2-3 Restriction: 0=none, 1=odd days, 2=even days
4-5 Schedule type: 0=weekly, 1=single-run, 2=monthly, 3=interval
6 Start-time type: 0=repeating, 1=fixed
7 Date range enable

Days Encoding

  • Weekly: days0 = weekday bitmask (bit0=Mon, bit1=Tue, … bit6=Sun)
  • Interval: days0 = remainder, days1 = interval in days
  • Monthly: days0 = day of month (1–31, 32=last day)

Start-time

  • Minutes from midnight (0–1439)
  • Sunrise: bit14 set, bit12=sign, bits 0–11=offset
  • Sunset: bit13 set, bit12=sign, bits 0–11=offset

Example

User says: "Water the lawn (station 0) and flower beds (station 1) every morning at 6 AM, 15 and 20 minutes, Mon–Fri, with weather adjustment"

{
  "pid": -1,
  "name": "Morning Watering",
  "v": "[67,31,0,[360,-1,-1,-1],[900,1200,0,0,0,0,0,0]]"
}
  • flag=67 (enabled + weather + fixed start times)
  • days0=31 (Mon–Fri)
  • start0=360 (6:00 AM)
  • dur0=900 (15 min), dur1=1200 (20 min)

Sensor-Based Program Adjustments

Automatically scale watering duration based on live sensor readings.

Adjustment Types

Type Name Behavior
1 LINEAR Linear interpolation: sensor min→max maps to factor1→factor2
2 DIGITAL_MIN If sensor ≤ min: factor1, else: factor2
3 DIGITAL_MAX If sensor ≥ max: factor2, else: factor1
4 DIGITAL_MINMAX If sensor ≤ min OR ≥ max: factor1, else: factor2

Example: Reduce Watering as Soil Gets Wetter

{
  "nr": 1,
  "type": 1,
  "sensor": 1,
  "prog": 0,
  "min": 10,
  "max": 90,
  "factor1": 2.0,
  "factor2": 0.0,
  "name": "Moisture Adjustment"
}

At 10% moisture → 200% watering. At 50% → 100%. At 90% → 0% (skip).

Monitor System

Monitors watch sensor values and trigger/block programs based on conditions.

Monitor Types

Type Name Description
1 MIN Trigger if sensor ≤ threshold
2 MAX Trigger if sensor ≥ threshold
3 SENSOR12 Read digital rain/soil sensor
10 AND Logical AND of up to 4 monitors
11 OR Logical OR of up to 4 monitors
12 XOR Logical XOR of up to 4 monitors
13 NOT Logical NOT of one monitor
14 TIME Time window (weekday + time range)
100 REMOTE Monitor on another OpenSprinkler

Example: Water Only When Dry AND During Morning Hours

Monitor A (MIN):  sensor=moisture, value1=30 → triggers when <30%
Monitor B (TIME): from=0600, to=1000, wdays=127 → 6–10 AM every day
Monitor C (AND):  monitor1=A, monitor2=B → triggers program only when both met

MCP Resources

The MCP server provides two dynamic resources that AI assistants can read for context:

Resource URI Content
api-overview opensprinkler://api-overview Complete API reference: program encoding, sensor types, adjustments, monitors
controller-summary opensprinkler://controller-summary Live data: firmware, stations, programs, status, WiFi

These resources give the AI all the documentation it needs to understand the system and create programs, configure sensors, and set up adjustments/monitors correctly.
